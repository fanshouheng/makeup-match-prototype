# 小红书平台支持修改计划

## 目标

在不改变浏览器本地人脸分析和匹配算法的前提下，让已授权创作者可以选择抖音或小红书作为主平台，并在匹配结果中打开对应平台主页和代表内容。

## 范围与假设

- 每位创作者在当前版本只绑定一个主平台。
- 抖音和小红书创作者进入同一个匹配池，用户匹配前不选择平台。
- 结果卡显示平台名称，但平台不会写入现有匿名产品事件。
- 同一创作者同时展示多个平台账号不在本次范围内；需要时再增加 `creator_channels` 子表。
- 小红书照片、面部特征和代表内容仍必须经过创作者本人或授权代表明确授权。

## 修改步骤

1. 数据库兼容迁移
   - 为 `creator_submissions` 和 `creators` 增加 `platform`、`profile_url`。
   - 将现有记录回填为 `douyin`，保留 `douyin_url` 兼容旧页面和旧函数。
   - 按平台校验主页和代表内容域名。
   - 更新 `approve_creator_submission`，复制平台字段并保持仅 `service_role` 可执行。
2. 提交与审核
   - 公开提交和管理员代录增加平台选择。
   - `submit-creator`、`admin-review` 同时兼容旧 `douyinUrl` 和新 `platform/profileUrl` 请求。
   - 重复检查改为 `platform + profile_url`。
3. 公开结果
   - 前端领域模型改用通用 `profileUrl`。
   - 结果卡显示平台名称，并打开对应平台主页。
4. 文档与验证
   - 更新 README、Supabase 设置和管理员审核 SOP。
   - 增加抖音、小红书、平台与域名不匹配的测试。
   - 运行单元测试、生产构建和桌面/移动端浏览器检查。

## 上线顺序

1. 通过 Supabase MCP 应用数据库迁移并运行安全、性能 Advisors。
2. 通过 Supabase MCP 部署 `submit-creator` 和 `admin-review`。
3. 部署前端。
4. 验证旧抖音数据、抖音新申请和小红书新申请。
5. 只在以上验证通过后导入已授权的小红书创作者。

## 完成标准

- 现有抖音创作者仍能读取、匹配和打开链接。
- 两个提交入口都能接受合法小红书链接，并拒绝平台与域名不匹配的链接。
- 审批后平台和主页字段完整进入公开创作者表。
- 匹配排序不按平台改变，结果卡不会重复同一创作者。
- 不新增用户照片、面部特征、创作者名称或具体链接遥测。
